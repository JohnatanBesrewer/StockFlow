SQLite по-простому, заново.
Давай начнём писать репозиторий заново, мало-по-малу, по шагам, от простого к сложному.
В моей программе будет следующая структура таблиц:

Таблица products:
- product_id - id продукта, тип UUID, является первичным ключом и уникальным значением, не может быть пустым;
- name - имя товара, тип str, не может быть пустым;
- unit - единицы измерения продукта, пользовательский тип _Unit(StrEnum) (но в БД тип str?), не может быть пустым;
- created_at: - дата+время создания записи в БД, тип datetime, не может быть пустым.

Таблица barcodes:
- barcode - бар-код продукта, тип str, является первичным ключом и уникальным значением, не может быть пустым;
- product_id - id продукта, тип UUID, внешний ключ, ссылается на первичный ключ таблицы products, не может быть пустым;
- created_at: - дата+время создания записи в БД, тип datetime, не может быть пустым.

Таблица sale_prices:
- price_id - id записи, тип UUID, является первичным ключом и уникальным значением, не может быть пустым;
- product_id - id продукта, тип UUID, внешний ключ, ссылается на первичный ключ таблицы products, не может быть пустым;
- price - собственно цена, тип Decimal, но в БД integer (домножаем на 100), не может быть пустым;
- valid_from - начало периода действия цены, тип datetime, не может быть пустым;
- valid_to - конец периода действия цены, тип datetime, если пустой, то цена актуальная.

Таблица suppliers:
- supplier_id - id записи, тип UUID, является первичным ключом и уникальным значением, не может быть пустым;
- name - собственно, имя, тип str, является уникальным значением, не может быть пустым;
- created_at: - дата+время создания записи в БД, тип datetime, не может быть пустым.

Таблица transactions:
- transaction_id - id записи, тип UUID, является первичным ключом и уникальным значением, не может быть пустым;
- transaction_type - тип транзакции (приход, продажа, списание, возврат инвентаризация итд), пользовательский тип _TransactionType(StrEnum) (но в БД тип str?), не может быть пустым;
- total_amount - общая сумма стоимости позиций транзакции, тип Decimal, но в БД integer (домножаем на 100), не может быть пустым;
- discount - скидка, тип Decimal, но в БД integer (домножаем на 100), используется только в SALE, в остальных случаях только "0",не может быть больше total_amount;
- supplier - поставшик, тип str, используется только в PURCHASE и RETURN_TO_SUPPLIER, в остальных случаях None
- comment - комментарий (опционально), тип str, может быть пустым;
- transaction_date - дата транзакции, тип datetime.date, не может быть пустым;
- created_at: - дата+время создания записи в БД, тип datetime, не может быть пустым.

Таблица transaction_items:
- item_id - id записи, тип UUID, является первичным ключом и уникальным значением, не может быть пустым;
- transaction_id - id транзакции, тип UUID, внешний ключ, ссылается на первичный ключ таблицы transactions, не может быть пустым;
- product_id - id продукта, тип UUID, внешний ключ, ссылается на первичный ключ таблицы products, не может быть пустым;
- product_barcode - штрих-код продукта, который был отсканирован, тип str, не может быть пустым;
- product_name - имя продукта на момент совершения транзакции, тип str, не может быть пустым;
- purchase_price - цена поступления за единицу продукта, тип Decimal, но в БД integer (домножаем на 100), может быть пустым в зависимости от типа транзакции;
- sale_price - цена продажи за единицу продукта, тип Decimal, но в БД integer (домножаем на 100), может быть пустым в зависимости от типа транзакции;
- product_quantity - количество продукта, тип integer, не может быть пустым;
- amount - общая стоимость одной позиции (product_price * product_quantity), тип Decimal, но в БД integer (домножаем на 100), не может быть пустым;
